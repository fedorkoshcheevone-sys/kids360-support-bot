# Часть 2 — Проектирование ветки «Возврат / двойное списание»

> Глубокая проработка AI-ветки Penny 2.0: ветвящаяся карта диалога, разбор системного промпта,
> метрики автоматизации и углублённые edge cases. Всё — на реальной базе знаний Kids360, без
> выдуманных фактов сверх неё.

---

## 1. Карта диалога (ветвящаяся)

Основной путь — **форма-мастер на кнопках** (детерминированная машина состояний, без LLM на каждом
шаге). Свободный текст доступен всегда как запасной вход: LLM извлекает из него поля формы и возвращает
пользователя в нужный шаг — он «переводчик» текста в состояние, а не собеседник. **LLM вызывается ровно
в двух точках:** vision (чтение чека) и разбор свободного текста. Ключевые решения — на **S1**
(тип причины) и **S2** (канал).

**S0. Вход (форма-мастер).** Пользователь попадает в ветку возврата по кнопке в дереве поддержки —
тема известна. Penny сразу начинает форму-мастер: «Помогу оформить заявку на возврат». Свободный текст
доступен всегда как запасной вход — LLM извлекает поля и возвращает в нужный шаг формы (не создаёт
параллельный чат).

**S1. Эмпатия + классификация причины.** Бот коротко по-человечески реагирует и определяет ТИП
проблемы (от типа зависит, возможен ли возврат в принципе):
- **(a)** двойное / повторное списание;
- **(b)** списание после отмены;
- **(c)** ошибочный / незнакомый платёж;
- **(d)** возврат за неиспользованное время — по условиям Kids360 (п. 11.13), как правило, НЕ
  возвращается; бот должен честно это сказать, но не путать с (a)–(c), которые под этот пункт НЕ подпадают.

→ Развилка: если **(d)** — честно объяснить про неиспользованное время и предложить отмену;
если **(a)–(c)** — продолжить к возврату (S2).

**S2. Канал подписки.** Бот выясняет: App Store / Google Play / оператор.
- **App Store** → возврат только через Apple (reportaproblem.apple.com); Kids360 сопровождает и
  подтверждает операцию, но не возвращает напрямую.
- **Google Play** → возврат через Google Play / поддержку Kids360.
- **Оператор** → через оператора.

Если пользователь не знает канал — бот помогает определить: «на каком устройстве оформляли —
iPhone или Android?»

**S3. Детали.** Сколько списаний, суммы, примерные даты.

**S4. Чек.** Направление к ЯВНОЙ кнопке загрузки (фикс UX). Развилка: пользователь может приложить
чек → S5; не может / нет доступа → сбор данных текстом + пометка для оператора.

**S5. Распознавание (vision).** Извлечение: сумма, дата, назначение, банк / канал, confidence. Развилка:
- confidence высокий → S6;
- низкий / не чек → просьба переснять ИЛИ эскалация с пометкой «чек нечитаем»;
- на чеке видно, что списание одно (а жаловались на два) → бот вежливо уточняет расхождение, не спорит.

**S6. Подтверждение.** Бот показывает распознанное («вижу N ₽ от DD.MM, назначение — Kids360, верно?»).
Развилка: нет → уточнить / перезапросить.

**S7. Сверка с биллингом → заявка на апрув.** Бэк **детерминированно (без LLM)** сверяет заявленное
списание с биллингом Kids360 по `account_id` (в приложении пользователь авторизован, id передаётся
автоматически): реально ли по аккаунту прошло два списания. AI формирует готовый **вердикт** и
структурированную заявку (тип причины, канал, данные чека, что подтвердил биллинг). Развилка по факту:
- **≥2 списания** → «двойное списание подтверждено» → рекомендация «возврат»;
- **1 списание** → «на ручную проверку» оператору.

Оператор получает готовую карточку и только **подтверждает или отклоняет** — не разбирается с нуля.
Бот НЕ обещает возврат. Апрув остаётся за человеком, потому что речь о деньгах.

> **В прототипе** биллинг — детерминированный **демо-мок** (`app/billing.py`, эндпоинт `POST /api/verify`),
> реальной интеграции с биллингом Kids360 / стор-API нет. В боевой версии здесь были бы запросы к биллингу
> по account_id и к App Store Server API / Google Play Developer API по каналу. Это **третья
> детерминированная точка** (не LLM): деньги не отдаём на откуп генерации.

**Сквозные переходы (из любого состояния):** запрос живого оператора → эскалация с уже собранным
контекстом; агрессия / стресс → усиленная эмпатия, без обещаний; не-своя тема → мягкий возврат к
сценарию или эскалация.

> **Ключевое отличие от старого бота:** старый бот на S1 сразу отказывал по п. 11.13 и на S2
> спрашивал банк карты; новый — сначала классифицирует причину и выясняет канал, а отказ по 11.13
> применяет ТОЛЬКО к типу (d).

## 2. Где живёт логика: форма, 2 LLM-точки и правила

Тема «возврат» известна уже на входе (пользователь нажал пункт в дереве поддержки) — значит нужна
быстрая процедура, а не беседа. Поэтому диалог — **детерминированная форма-мастер**, а LLM подключается
точечно. Что где решается:

| Что | Где решается | LLM? |
|---|---|---|
| Порядок вопросов, тексты, ветвление (причина → канал → чек → подтверждение) | Детерминированная форма-мастер (фронт) | нет |
| Понимание свободного текста → поля формы | `INTERPRET_PROMPT` — строгий JSON, низкая температура | **да** |
| Чтение чека (сумма / дата / канал) | vision `RECOGNIZE_PROMPT` | **да** |
| Сверка двойного списания + вердикт | Правила по биллингу (`/api/verify`, демо-мок) | нет |
| Апрув возврата | Человек-оператор | — |

Принципы старого промпта никуда не делись — но теперь они **гарантированы конструкцией**, а не
«надеждой на генерацию»:

- **Сначала причина и канал, потом ответ про возврат** — порядком шагов формы (детерминированно);
  отказ по п. 11.13 применяется только к типу «за неиспользованное время». Старый бот отказывал по 11.13
  без выяснения причины — здесь это в принципе невозможно.
- **Никогда не обещать возврат** — зашито в тексты формы и в промпт разбора; решение и апрув — за оператором.
- **Не верить на слово о двойном списании** — сверка с биллингом Kids360 по `account_id` (правила, не генерация).
- **Грунтинг, без данных карты, эскалация с контекстом** — разбор свободного текста грунтуется на базе знаний
  и не просит CVV/пароли; просьба о человеке из любого шага → передача оператору с уже собранными полями.

Промпты — в [`app/prompts.py`](../app/prompts.py) (`INTERPRET_PROMPT`) и [`app/llm.py`](../app/llm.py)
(`RECOGNIZE_PROMPT`); правила биллинга — [`app/billing.py`](../app/billing.py); поведенческая спецификация —
`SYSTEM_PROMPT`.

> LLM намеренно ограничен ровно двумя задачами — **прочитать чек** и **понять свободный текст** — именно
> чтобы не словить три типичных провала LLM в поддержке: **галлюцинация фактов**, **обещания сверх
> полномочий**, **соскальзывание в не-свою тему**. Всё, что касается денег, — детерминированно.

## 3. Метрики успеха ветки

Как измерять, что автоматизация работает. Ориентиры — гипотеза, уточняется на данных.

| Метрика | Что показывает | Как считаем | Ориентир |
|---|---|---|---|
| **Deflection rate** (доля автозакрытых) | Часть обращений по возврату, закрытых ботом без оператора | Закрытые ботом / все обращения ветки | Гипотеза: рост к 40–60% на типовых кейсах |
| **Точность распознавания чека** | Верно ли извлечены сумма / дата / канал | Доля чеков с корректными полями (ручная сверка на выборке) | Гипотеза: ≥ 90% на читаемых чеках |
| **Корректность классификации причины** | Верно ли кейс отнесён к (a)–(d) | Сверка на выборке | Критично: от этого зависит, не откажут ли ошибочно |
| **Доля ложных эскалаций** | Бот эскалировал то, что мог закрыть сам | Ложные эскалации / все эскалации | Минимизируем, но осторожно |
| **Доля ложных автозакрытий** (критичная) | Бот закрыл то, что требовало человека | Выборочный аудит | Держать около нуля — цена ошибки высока |
| **CSAT по ветке** | Оценка родителя после диалога | Опрос в конце | Не должен падать из-за автоматизации |
| **Время до первого осмысленного ответа** | Скорость vs старый кнопочный флоу | Замер по логам | Заметно быстрее |
| **Точность вердикта по биллингу** | Верно ли AI определил двойное списание (совпадение с фактом) | Сверка вердикта с реальными списаниями на выборке | Критично: от этого зависит правильность апрува |

> **Приоритет:** в детском платёжном сервисе «доля ложных автозакрытий» и «корректность
> классификации» важнее, чем максимальный deflection. Лучше передать человеку лишнее, чем ошибочно
> отказать в возврате. Это осознанный продуктовый выбор.

## 4. Edge cases (углублённые)

Формат: ситуация → риск → решение (в форме / правилах / логике).

1. **Сразу шлёт чек без текста.** Риск: бот не знает канал / причину. Решение: после распознавания
   всё равно уточняет канал и тип причины, не пропускает классификацию.
2. **Чек нечитаемый / не чек (скрин игры, фото экрана).** Риск: бот распознает мусор. Решение: при
   низком confidence не додумывает — просит переснять или эскалирует «чек нечитаем»; на «не чек»
   вежливо поясняет, что нужен документ об операции.
3. **Эмоциональный / агрессивный пользователь.** Риск: бот либо обещает возврат (нельзя), либо
   холоден. Решение: усиленная эмпатия + честность без обещаний; при агрессии / просьбе человека —
   из любого шага передача оператору с уже собранным контекстом.
4. **Канал — App Store.** Риск: бот технически не может вернуть. Решение: честно объясняет про Apple
   (reportaproblem.apple.com), но фиксирует обращение и подтверждает операцию — сопровождает, а не «отфутболивает».
5. **Возврат за неиспользованное время (тип d).** Риск: пользователь ждёт возврат, а по п. 11.13 не
   положено. Решение: честно и мягко объясняет ограничение, предлагает отмену, не создаёт ложных
   ожиданий; но проверяет, точно ли это (d), а не замаскированное двойное списание.
6. **OpenAI / vision недоступен (сбой API).** Риск: бот «падает» молча. Решение: graceful degradation —
   сообщает, что распознавание временно недоступно, просит описать чек текстом (сумма / дата / канал)
   или эскалирует; диалог не рвётся.
7. **Чек на иностранном языке / другая валюта.** Риск: неверный парсинг. Решение: vision извлекает
   как есть + помечает валюту; если валюта не ₽ — бот уточняет и передаёт оператору с пометкой.
8. **Клиент заявил два списания, а биллинг подтверждает одно.** Риск: авто-отказать правому или
   ошибочно вернуть. Решение: сверка с биллингом даёт вердикт «на ручную проверку» (needs_review) —
   не авто-решаем; оператор получает и чек, и данные биллинга и решает сам. Бот не спорит с клиентом.
